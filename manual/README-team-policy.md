# 🚨 GPU 컨테이너 사용 정책 — 금지 행위 및 주의사항

> **배경:** 2026-07-03 발생한 team03 컨테이너 마비 사고를 계기로 전 팀 공통 운영 정책을 안내합니다.
> 아래 행위는 **본인 팀뿐 아니라 같은 서버를 쓰는 다른 팀 전체**에 피해를 줄 수 있어 금지합니다.

---

## 1. 개요 (무슨 일이 있었나)

- 2026-07-03, **team03 컨테이너가 SSH 로그인·명령 실행 등 모든 작업이 불가능한 상태로 마비**되었습니다.
- 원인: 컨테이너 내부에서 만든 웹앱(`localhost:8123`)을 외부에 노출하려고, **외부 터널 도구 `cloudflared`를 자동 재시작 스크립트(`watchdog.sh`, 60초 루프)로 상시 구동**한 것입니다.
- 이 스크립트가 이전 터널 프로세스를 종료하지 않고 계속 새로 띄우면서, **38일간 cloudflared 프로세스가 13,000개 이상 누적**되었습니다.
- 결과적으로 컨테이너의 **프로세스 생성 한도가 소진**되어 새 프로세스를 만들 수 없게 되었고(`fork: Resource temporarily unavailable`), 로그인·명령·복구 시도가 전부 실패했습니다.
- 관리자가 컨테이너를 **강제 재시작**하여 복구했습니다. (`/workspace` 등 저장 데이터는 보존되었으나, 실행 중이던 모든 세션과 미저장 상태는 유실되었습니다.)
- 추가로, 같은 컨테이너에서 **또 다른 외부 터널(`ngrok`)** 과 다수의 상시 서비스가 방치되어 있던 것도 확인되었습니다.

## 2. 이게 초래하는 영향

| 대상 | 영향 |
|---|---|
| **본인 팀** | 컨테이너 전면 마비, 접속 불가, 진행 중이던 학습·세션·미저장 작업 유실, 관리자 강제 재시작으로 환경 초기화 |
| **같은 서버의 다른 팀** | 호스트의 공유 자원(메모리·프로세스/스레드·파일디스크립터) 잠식으로 **동반 성능 저하 및 장애 위험** |
| **서버 전체** | GPU 서버 안정성 저하, 관리자 긴급 개입 필요 |
| **보안** | 학과 GPU 서버의 내부 서비스가 **인증 없이 인터넷에 노출** → 외부 공격 표면 및 데이터 유출 경로 발생 |

## 3. 위험 요소 (무엇이 왜 위험한가)

- **외부 터널링 도구** — `cloudflared`, `ngrok`, `frp`, `localtunnel`, `bore`, `tailscale funnel` 등
  - 컨테이너 안에서 **바깥으로 먼저 연결을 맺는 방식**이라, 관리자가 설정한 방화벽·포트 통제를 **우회**합니다. 열린 포트가 없어도 내부 서비스가 공개 URL로 노출됩니다.
- **자동 재시작 / 무한 루프 스크립트** — `watchdog`류, `while true`, `nohup ... &`, 자체 cron 등
  - 프로세스가 종료되지 않고 누적되면 **자원 누수 → 컨테이너·서버 마비**로 이어집니다. 이번 사고의 직접 원인입니다.
- **상시 서비스 데몬의 방치·중복 실행** — 웹서버, 노트북 서버 등
  - 정리되지 않은 프로세스가 쌓여 자원을 잠식합니다.
- **세션 미정리** — `screen`, `tmux`, `vscode-server`, 각종 에이전트 등
  - 사용을 마친 세션이 수십 개씩 남으면 마찬가지로 자원을 소모합니다.

## 4. 앞으로 금지 조항

> 아래 행위는 **금지**합니다. 위반 시 컨테이너가 정지·초기화될 수 있으며, 반복 시 이용이 제한될 수 있습니다.

- [ ] **① 외부 터널링 도구 사용 금지** — cloudflared·ngrok·frp·localtunnel·bore·tailscale funnel 등, 컨테이너 내부 서비스를 외부 인터넷에 노출하는 일체의 터널링.
- [ ] **② 자동 재시작·무한 루프로 프로세스 반복 생성 금지** — watchdog식 respawn, `while true`로 데몬 재구동 등. 모든 프로세스는 단일 인스턴스로 실행하고 종료를 관리할 것.
- [ ] **③ 상시 외부 서비스 호스팅 금지** — 컨테이너는 학습·연구·개발 용도이며, 외부 공개용 웹서비스를 상주시키지 말 것.
- [ ] **④ 백그라운드 프로세스 방치 금지** — `nohup`/`&`로 띄운 작업은 끝나면 반드시 종료. 노트북·웹서버도 쓰지 않을 때 내릴 것.
- [ ] **⑤ 자원 한도(프로세스·메모리·쿼터) 우회 시도 금지.**
- [ ] **⑥ 세션 정리 의무** — 사용하지 않는 screen/tmux/vscode-server/에이전트 세션은 종료할 것.

## 5. 허용 · 권장 대안 (웹앱을 확인하고 싶다면)

- **본인 PC에서 미리보기 → SSH 로컬 포트포워딩 사용**
  외부에 공개하지 않고, 이미 인증된 SSH 연결을 통해 **본인 컴퓨터에서만** 접속하는 안전한 방식입니다.
  ```bash
  # 예) 컨테이너의 8123 웹앱을 내 PC의 localhost:8123으로 연결
  #     (접속 경로/포트/bastion 여부는 관리자 안내에 따름)
  ssh -N -L 8123:localhost:8123 -p 22023 team03@<서버주소>
  # 이후 브라우저에서 http://localhost:8123 접속
  ```
- **팀 외부 사람에게 공개해야 하는 경우** — 임의로 터널을 열지 말고 **관리자에게 요청**하세요. 승인된 절차로 안전하게 열어 드립니다.

### (참고) 관리자 측 후속 조치

- 컨테이너별 **프로세스 수 상한(pids limit)** 적용 — 한 팀의 폭주가 서버 전체로 번지지 않도록.
- **프로세스 급증 모니터링 알림** 추가.
- 컨테이너 **아웃바운드(egress) 트래픽 통제·모니터링** 검토 — 터널링 차단.
